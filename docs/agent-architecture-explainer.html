Agent Architecture, Explained Simply

Think of the app like a small company that works on datasets. We cleaned up the workers, the skills they use, and the way they pass work to each other.

Agents

The workers with roles, like analyst and engineer.

Skills

The reusable abilities those workers use to do the job.

Services

The fact machines that compute profiles, warnings, and readiness.

Coordinator

The traffic police deciding who should lead the conversation.

1. We separated workers from skills

Before, some files called “agents” were really just abilities mixed into product code. Now the split is cleaner:

Agents = workers

  • DataAnalystAgent = the analyst worker
  • DataEngineerAgent = the engineer worker

Skills = abilities

  • QuerySkill = answer analysis questions
  • ReportingSkill = make reports
  • EngineeringSkill = handle cleaning and readiness

Simple version: worker = person, skill = what that person knows how to do.

2. The two main workers can now take turns

The app now has three key pieces:

  • DataAnalystAgent
  • DataEngineerAgent
  • TeamCoordinator

The TeamCoordinator is like a teacher in class deciding who should answer.

  • “What is the most important feature?” → analyst answers
  • “Can we trust these columns?” → engineer answers
  • “Okay, what insight next?” → analyst takes over again

So now the chat can switch between roles instead of one agent pretending to know everything.

3. We made DataEngineerAgent smaller and cleaner

Before, the engineer role was carrying too many responsibilities at once:

  • deciding when to speak
  • building the engineering contract
  • making prompts
  • answering cleaning questions
  • handling profile logic

That was too much for one class, so we extracted the reusable brain into EngineeringSkill.

  • DataEngineerAgent = the engineer role
  • EngineeringSkill = the engineer ability/brain

4. Your profiling script still matters a lot

We did not throw away your profiling logic. These app services still compute the truth:

  • backend/app/services/profiling.py
  • backend/app/services/data_engineering.py

They still do the real fact work:

  • column types
  • missing values
  • correlations
  • readiness score
  • warnings
  • semantic roles

So the split is:

  • services = compute facts
  • skills = use the facts
  • agents = talk to the user

5. We made the code more reusable

Before, too much code was glued directly to the local database, local files, local query engine, and local artifact storage. Now there is a cleaner plug-and-socket system.

Contracts

These are the rules that say what the core package needs.

  • ProfileProvider
  • QueryRunner
  • ArtifactStore
  • DatasetContext

Adapters

These are the real local implementations used by this app.

  • load from SQLite
  • run SQL with DuckDB
  • save charts and reports locally

So the reusable code now says “give me a query runner” instead of “I only work with this one exact local function”.

6. We created a reusable core package

backend/data_berge_core

  • contracts
  • reusable skills
  • runtime-neutral pieces

backend/app

  • FastAPI routes
  • local DB and storage adapters
  • local settings
  • role agents
  • workflows

The easiest mental picture is:

  • core package = engine
  • app package = car body around the engine

7. The folder structure is cleaner now

Agents folder

  • data_analyst.py
  • data_engineer.py
  • coordinator.py
  • team.py

Core skills folder

  • intake.py
  • profiler.py
  • query.py
  • visualization.py
  • reporting.py
  • governance.py
  • engineering.py

8. What happens now when a user chats?

1
Upload happens.
2
Profiling runs: schema, stats, missingness, and relationships.
3
Engineering contract runs: roles, warnings, readiness, and suggested actions.
4
User asks something in chat.
5
The coordinator chooses who should lead: analyst or engineer.
6
That agent uses its skill set to answer.
7
The answer comes back to the user.

9. Why this is better

Because now the app is less confusing and easier to grow.

  • roles are clearer
  • skills are reusable
  • services still compute the truth
  • chat can hand off between roles
  • code is easier to extend later

10. The shortest mental model

  • services = calculators
  • skills = abilities
  • agents = workers
  • coordinator = traffic police
  • core package = reusable brain
  • app package = local app wiring

That’s basically the whole story.
